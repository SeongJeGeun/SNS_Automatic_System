# MindFactory SNS Automation System Runbook

본 문서는 마인드팩토리 SNS 자동화 시스템(Threads, Instagram, Telegram, Google Sheets 연동형 멀티 에이전트)의 안정적인 무인 공정 운영을 보장하기 위한 **장애 탐지, 장애 대응, 검증 절차 및 수동 개입 기준 매뉴얼**입니다. 운영자는 이 문서의 절차에 따라 시스템을 유지관리하고 대응해야 합니다.

---

## 🔄 1. 시스템 아키텍처 및 상태(State) 기계

### 📊 플랫폼별 역할 분담
* **Threads**: 가벼운 대화형 구어체 팩폭(`~죠?`, `~요`) 기반의 1차 훅 및 토론 유도 채널.
* **Instagram**: 단정체(`~입니다`, `~하세요`) 기반의 시각적 요약 정보 및 소장/저장(Saves), DM 공유(Sends) 최적화 카드뉴스 채널.
* **Telegram**: 시스템을 원격으로 감시하고 즉시 실행 및 수동 개입을 제어하는 실시간 양방향 게이트.
* **Google Sheets**: 전체 계정 성장 원장, 포스팅 실적 및 장애 로그를 보존하는 마스터 데이터베이스.

### 🤖 에이전트별 역할 및 정의
1. **Topic Agent**: 주간 RAG 분석으로 핵심 주제 도출.
2. **Threads Agent**: 1~2줄 강한 훅 포스트 기획.
3. **Instagram Agent**: 1:3 꿀팁/공감 황금 비율 기반 7장 카드뉴스 렌더링 대본 및 캡션 작성.
4. **Engagement Agent**: 댓글 수집 및 Q&A 분류.
5. **Self-Healing Agent**: 오류 발생 시 백오프 리트라이 및 프롬프트 보정.
6. **Telegram Interface**: 명령어 처리 및 상태 회신봇.
7. **Spreadsheet Reporter**: 지표 원장 기록기.

### 🧬 콘텐츠 상태(State) 값 정의
콘텐츠 생명 주기에 따라 각 컨테이너는 아래 8가지 상태값을 가지며, 시스템 상태 파일(`agent_runs/agent_status.md` 및 `manifest.json`)에 기록됩니다.

| 상태 | 정의 | 전이 조건 |
| :--- | :--- | :--- |
| **`draft`** | 최초 대본 및 이미지가 생성된 임시 상태 | 콘텐츠 기획/렌더링 완료 즉시 전이 |
| **`approved`** | 인간 관리자가 최종 게재를 승인한 상태 | 텔레그램/수동 승인 신호 수신 시 |
| **`scheduled`** | 업로드 대기 큐에서 예약 시각을 기다리는 상태 | 승인 후 업로드 스케줄러 등록 시 |
| **`published`** | API 최종 발행 완료 상태 | Threads/Instagram API 업로드 성공 응답 수신 시 |
| **`failed`** | 복구 실패 혹은 치명적 오류로 발행이 포기된 상태 | 리트라이 횟수 초과 및 API 영구 거절 시 |
| **`recovering`** | 자가 치유 에이전트가 자동 복구를 시도 중인 상태 | 에러 감지 후 Exponential Backoff 리트라이 진입 시 |
| **`recycle_candidate`** | 기준 성과(130%)를 초과하여 차주 재사용 대상이 된 상태 | 일일 성과 수집 분석 시 기준치 초과 감지 시 |
| **`escalated`** | 인간 관리자의 수동 조치 및 승인을 홀드 대기 중인 상태 | 2회 연속 자가 치유 실패 혹은 인증 수동 갱신 필요 시 |

## 🔄 3. 플랫폼별 업로드 및 검증 절차

### 🧵 Threads 업로드 검증 절차

Threads API의 발행 성공 응답값만을 맹신하지 않고, 실제 사용자의 Threads 프로필 피드에 게시물이 온전히 반영되었는지 정밀 검증하기 위한 다중 체크 절차입니다.

#### 1. API 기반 확인 절차
* **[Step 1-1] 선결 검증 및 토큰 상태 확인**
  - 업로드 API 호출 전 `GET https://graph.threads.net/v1.0/me`를 호출하여 API 액세스 토큰의 유효성을 점검합니다.
* **[Step 1-2] 미디어 컨테이너 상태 점검 (Status Polling)**
  - 텍스트/미디어 업로드 후 생성된 컨테이너 ID(`threads_container_id`)를 활용하여 상태를 조회합니다:
    ```
    GET https://graph.threads.net/v1.0/<threads_container_id>?fields=status,error_message
    ```
  - 응답 JSON의 `status` 필드가 반드시 `"FINISHED"` 상태인지 대기/확인합니다. 만약 `"ERROR"`나 `"EXPIRED"` 상태인 경우 다음 단계 진입을 차단합니다.
* **[Step 1-3] API publish 최종 응답 및 미디어 ID 추출**
  - 컨테이너 발행 API(`POST https://graph.threads.net/v1.0/me/media_publish?creation_id=<threads_container_id>`) 호출 결과로 반환되는 200 OK 응답 및 생성된 포스트 ID(`threads_post_id`)를 파싱하여 정상 반환 여부를 확인합니다.
* **[Step 1-4] API를 통한 게시물 데이터 조회 검증 (Deep Verification)**
  - 생성된 `threads_post_id`를 사용하여 게시물 메타데이터를 다시 읽어옵니다:
    ```
    GET https://graph.threads.net/v1.0/<threads_post_id>?fields=id,media_product_type,permalink,text,timestamp
    ```
  - 본문(`text`) 내용의 첫 10바이트가 기획서와 일치하는지, `permalink` 값이 유효한 URL 포맷인지 정밀 체크하여 API 통신 레벨의 최종 성공을 확정합니다.

#### 2. 수동 확인 절차 (운영자 실검증)
* **[Step 2-1] 공개 프로필 피드 반영 여부 실시간 확인**
  - 모바일 Threads 앱 또는 웹 브라우저(`https://www.threads.net/@mind_factory_us`)에 직접 접속합니다.
  - 최상단 피드에 방금 업로드된 훅 텍스트가 정상적으로 표시되는지 육안 확인합니다.
* **[Step 2-2] 미디어 렌더링 및 레이아웃 검수**
  - 텍스트 내 줄바꿈이나 이모지가 깨지지 않았는지, 이미지/비디오가 첨부된 경우 찌그러지거나 흐리게 표현되는 리소스 렌더링 이상 여부를 진단합니다.
* **[Step 2-3] 게시물 상세 정보 및 활동 보기 기능 작동 여부**
  - 업로드된 게시물을 클릭하여 상세 뷰로 진입이 가능한지 확인합니다.
  - 좋아요, 답글 쓰기 창이 활성화되어 있으며, 타 계정으로 접근했을 때 댓글 작성이 정상적으로 열려 있는지 테스트 댓글을 1회 달아 검증합니다.

#### 3. 업로드 실패 시 점검 순서
1. **API 오류 코드 분석**: 응답으로 반환된 Graph API Error Code(예: 429 - Rate limit, 190 - Invalid OAuth Token 등)를 추출합니다.
2. **네트워크 및 라우팅 상태 점검**: 로컬 및 클라우드 호스팅 서버의 DNS 확인 및 Meta API 엔드포인트 대상의 Ping/Traceroute 상태를 체크합니다.
3. **로컬 캐시 및 세션 데이터 정화**: 토큰 또는 컨테이너 임시 정보를 파이프라인에서 지우고 캐시를 만료 처리합니다.
4. **점진적 재시도 실행**: 10초, 30초, 60초 백오프 간격으로 컨테이너 재생성부터 최대 3회 재시도를 실행합니다.
5. **에스컬레이션(Escalation) 전이**: 3회 시도가 모두 실패하면 상태를 즉시 `failed`로 변경하고 텔레그램을 통해 에러 덤프와 함께 수동 조치 알림을 발행합니다.

#### 4. Threads 검증 운영 체크리스트
* **[ ]** `GET /me` 검증에서 유효한 200 OK 응답을 확인했는가?
* **[ ]** 컨테이너 `status` 폴링의 최종 결과가 `"FINISHED"` 상태인가?
* **[ ]** 발급된 `threads_post_id`로 게시물 세부 데이터를 역조회하여 데이터 일치성을 검사했는가?
* **[ ]** `https://www.threads.net/@mind_factory_us` 프로필 피드 최상단에 실제 포스트가 정상 게재되었는가?
* **[ ]** 타 계정으로 해당 스레드에 진입하여 게시물 상세 보기 및 댓글(답글) 작성 창이 활성화되는가?

#### 5. 시점별 지표 모니터링 및 텔레그램 보고 체계
* **[ ] 1차 보고 (게시 직후)**:
  - 수집 데이터: `post_id`, `permalink`, `게시 시각`
  - 발송 형태: 텔레그램 즉시 발송 및 로컬 임시 JSON 캐싱
* **[ ] 2차 보고 (게시 24시간 뒤)**:
  - 수집 데이터: `조회수`, `좋아요 수`, `댓글 수`, `리포스트/인용 수`, `프로필 유입/팔로우 증가 여부`
  - 액션: Threads API를 통한 성과 수집 후 전날 수집 지표와 24h 성장 추이를 텔레그램 요약 발송
* **[ ] 3차 최종 보고 (게시 72시간 뒤)**:
  - 수집 데이터: 72시간 누적 성과 종합 지표
  - 피드백 액션: 72시간 데이터를 계정 5회 평균치와 비교 분석하여 **업로드 간격(PIPELINE_INTERVAL_SECONDS)** 단축/연장 및 차기 **RAG 생성 훅 문구의 어조** 강도를 자가 조율하는 스위치 스케줄링 진행.

### 📸 Instagram 카드뉴스 업로드 검증 절차
1. **임시 이미지 호스팅 검증**: 구글 드라이브 임시 폴더에 업로드된 `page1.png` ~ `page7.png`가 퍼블릭 읽기 가능한 URL로 정상 생성되었는지 확인합니다.
2. **자식 미디어 상태 검증**: 각 슬라이드 컨테이너 생성 후 `status_code`가 `FINISHED`가 될 때까지 최대 75초간 폴링을 검증합니다.
3. **부모 카러셀 최종 발행**: `media_publish` API가 `status_code 200`을 반환하는지 검증하고, 최종 `published_id`를 획득하여 리포팅 파일에 기록합니다.

### 📝 Google Sheets / Excel 보고서 생성 검증 절차
1. **시트 동적 탭 생성**: 오늘 날짜의 월 단위 탭(`M[X]`)이 자동으로 확인되거나 신규 생성되었는지 확인합니다.
2. **기록 실패 대비 캐시 저장**: 기록에 실패할 경우 `logs/daily_report_backup.json`에 원본 데이터를 보존한 후, 다음 배치 구동 시 병합하여 자동 기입을 처리합니다.

### 🔔 Telegram 알림 검증 절차
1. **메시지 전송 성공 여부**: `requests.post` 응답의 `status_code == 200` 및 JSON 응답의 `"ok": true` 여부를 검사합니다.
2. **대체 로깅**: 네트워크 전송 실패 시, 즉시 `logs/telegram_failures.log` 파일에 타임스탬프와 메시지 원본을 저장하여 추적할 수 있도록 합니다.

## 🚨 4. 장애 대응 런북 (Troubleshooting)

### 🔑 4.1 토큰 만료 (OAuthException / 401)
* **장애 감지**: API 호출 시 HTTP 401 오류 또는 응답 데이터 내 `Error validating access token` 감지.
* **즉시 조치**:
  1. `instagram_token_manager.py` 실행하여 저장된 리프레시 토큰으로 자동 갱신 시도.
  2. 구글 API의 경우 로컬 `token.json` 삭제 후 `authenticate_google.py` 수동 구동하여 재인증 진행.
* **재시도 정책**: 자동 갱신은 1회만 수행하며, 실패 시 에스컬레이션합니다.
* **에스컬레이션**: 텔레그램으로 `[토큰 만료 긴급 수동 인증 필요]` 메시지 발송 후 파이프라인 정지 (`escalated` 상태).

### 🚫 4.2 API 차단 및 Rate Limit (HTTP 429)
* **장애 감지**: HTTP 429 Too Many Requests 반환 또는 Instagram API로부터 Action Blocked(code: 4, subcode: 2207051) 수신.
* **즉시 조치**:
  1. `agent_runs/instagram_publish_cooldown.json` 파일을 생성하여 24시간 동안 모든 자동 업로드 루프를 물리적으로 차단합니다.
  2. 시스템은 즉시 `recovering` 상태로 전이합니다.
* **재시도 정책**: 24시간 쿨다운 종료 이후 1회 재발행을 시도합니다.
* **에스컬레이션**: 24시간 이후 재시도 시점에도 차단이 해제되지 않으면 `escalated`로 강등하고 무인 발행을 정지합니다.

### ❌ 4.3 게시 실패 (Publish Failure)
* **장애 감지**: 미디어 컨테이너 생성이 연속 실패하거나 부모 카러셀 최종 발행 시 예외 발생.
* **즉시 조치**:
  1. 업로드하려던 카드뉴스 이미지 경로 및 캡션의 바이트 크기를 진단합니다.
  2. 임시 구글 드라이브 다운로드 링크의 유효성 수동 테스트를 진행합니다.
* **재시도 정책**: 10초 간격으로 최대 3회 점진적 백오프(Exponential Backoff)를 시도합니다.
* **에스컬레이션**: 3회 최종 실패 시 상태를 `failed`로 변경하고 텔레그램에 에러 세부 덤프를 전송합니다.

### 🖼️ 4.4 이미지 렌더링 에러 (Rendering Error)
* **장애 감지**: `page*.png` 파일이 정상 생성되지 않았거나, 파일 크기가 0바이트인 현상 발생.
* **즉시 조치**: `generated_backgrounds/` 및 `static/` 경로의 리소스 및 템플릿 손상 여부 자동 진단.
* **재시도 정책**: 자가 치유 발전기(`self_healing_generator.py`)를 통해 대본 구조와 이미지를 1회 자동 재생성합니다.
* **에스컬레이션**: 재생성본 렌더링도 실패 시 파이프라인을 정지시키고 `failed` 처리합니다.

### 📣 4.5 Telegram 전송 실패
* **장애 감지**: `telegram_agent.py` 내 API 호출 타임아웃 혹은 Connection Error 발생.
* **즉시 조치**: 알림 서비스를 무알림 모드로 강제 전환하고 `logs/telegram_failures.log` 파일에 실패 내역을 적재합니다.
* **재시도 정책**: 백그라운드 큐를 이용해 5분 뒤에 1회 다시 보내기를 시도합니다.
* **에스컬레이션**: 계속 실패하는 경우 터미널 표준 에러(stderr)에 경고를 남기며, 메인 파이프라인 자체는 정상 진행(Silent Run)시킵니다.

### 📊 4.6 구글 시트 기록 실패
* **장애 감지**: `gspread.exceptions.APIError` 또는 구글 API 할당량 초과 에러 발생.
* **즉시 조치**: 업로드 완료 데이터를 로컬 백업 파일(`logs/daily_report_backup.json`)에 임시 기입(Append)하여 보존합니다.
* **재시도 정책**: 3시간 뒤 다음 파이프라인 루프가 돌아올 때 로컬 캐시 파일을 로드하여 시트에 병합 기입을 자동 재시도합니다.
* **에스컬레이션**: 3회 연속 누적 실패 시 텔레그램으로 `[구글 시트 동기화 장애]`를 통보합니다.

### 🔄 4.7 자가 치유 실패 (Self-Healing Failure)
* **장애 감지**: 자가 치유 전략 적용 이후 재생성된 대본마저 품질 기준을 미달하거나 API 업로드에 연속 실패하는 경우.
* **즉시 조치**: 시스템은 자동 복구 루프를 즉시 중단하고 홀드 모드로 진입합니다.
* **재시도 정책**: 추가 자동 재시도는 전면 금지합니다.
* **에스컬레이션**: 상태를 즉시 `escalated`로 전환하고, 텔레그램 챗방에 수동 조치 승인 링크 및 원인 요약 정보를 즉시 전송하여 수동 개입을 요구합니다.

## 🛠️ 5. Self-Healing 및 Fallback 메커니즘

```
[장애 감지] ➡️ [1차 자가 치유 (프롬프트 조정/백오프)] ➡️ [2차 실패 시 로컬 Fallback 캐싱] ➡️ [에스컬레이션 (인간 승인 전환)]
```

### 1. 자동 복구 조건 및 재시도 정책
* 직전 포스트의 조회수가 100회 미만일 경우 `need_healing` 플래그를 활성화하고 차기 콘텐츠 기획 시 비주얼 및 카피 다각화 프롬프트를 자동으로 주입합니다.
* API 에러 시 최대 3회 재시도를 진행하며, 각 재시도 간 대기 시간은 $10 \times (2^{\text{attempt}-1})$ 초로 점진적 증가시킵니다.

### 2. 로컬 Fallback 저장 방식
구글 시트 연동 차단이나 네트워크 장애 시 모든 지표는 로컬 `logs/daily_report_backup.json`에 다음의 포맷으로 누적 기록됩니다.
```json
[
  {
    "timestamp": "2026-05-28 00:50:23",
    "published_id": "180238491029",
    "title": "규율의 힘",
    "status": "cached"
  }
]
```

### 3. 인간 승인 전환 조건 (Human-in-the-Loop)
* 품질 에이전트(`content_evaluator.py`)가 판단한 품질 점수가 연속 2회 기준치 미달일 때.
* 3회 이상의 백오프 시도 후에도 플랫폼 API가 에러 상태를 유지할 때.
* 수동 상태 변경 명령인 `/pause`가 수신되었을 때.
* **승인 프로세스**: `escalated` 상태가 되면 텔레그램으로 `[Human Approval Needed]` 알림이 전송되며, 관리자가 `/approve` 또는 로컬에서 상태를 `approved`로 설정하기 전까지 모든 게시물 발행 스케줄은 전면 홀드됩니다.

## 📈 7. 핵심 지표 (KPI & Dashboard)

시스템은 일일 마스터 보고서 생성 시 다음의 핵심 지표(KPI)를 종합 산출하여 구글 시트와 텔레그램에 전송합니다.

1. **오늘 게시 수 (Daily Posts)**: 당일 성공적으로 업로드된 Threads 및 Instagram 총 게시물 개수.
2. **발행 성공률 (Publish Success Rate)**: $\frac{\text{성공한 업로드 수}}{\text{총 시도한 업로드 수}} \times 100$ (목표치: **98% 이상**).
3. **실패 수 (Failure Count)**: 자가 치유를 포함하여 최종 발행 실패(`failed`) 처리된 개수.
4. **재시도 수 (Retry Count)**: API 오류나 포맷 에러로 인해 자가 치유(Backoff) 및 재생성이 가동된 총 횟수.
5. **알림 성공 수 (Notification Delivery)**: 텔레그램을 통해 송신 성공한 알림의 비율 (목표치: **100%**).
6. **보고서 생성 여부 (Report Generated)**: 당일 아침 9시 구글 드라이브용 일일보고서 파일 생성 여부 (T/F).
7. **에스컬레이션 횟수 (Escalation Count)**: 인간 개입으로 이관된 횟수 (목표치: **주 0회**).
